# Book Translator (Script-Orchestrator)

Этот проект автоматизирует перевод больших текстовых файлов (ранобэ, манга, веб-романы) с любого языка на русский. Использует `gemini-cli` для многоэтапной обработки текста. Система спроектирована с упором на отказоустойчивость, параллелизацию и высокое качество итогового текста.

Поддерживаемые исходные языки: японский, корейский, китайский, английский и любой другой — достаточно создать стайлгайд.

## Основные возможности

* **Поддержка любого исходного языка**: Конвейер параметризован через языковые коды (`--source-lang`, `--target-lang`). Бандлированные стайлгайды: `ja_ru`, `ko_ru`, `zh_ru`, `en_ru`.
* **Интеллектуальное разделение глав**: Автоматически разбивает большие файлы на небольшие семантически связанные части (`chunk_*.txt`).
* **Четырёхэтапный конвейер**:
  1. **Поиск терминов (`discovery`)**: Параллельно обрабатывает все чанки для поиска новых терминов. Предлагает добавить их в основной глоссарий (с учётом стайлгайда и контекста мира).
  2. **Перевод со скользящим окном (`translation`)**: После подтверждения терминов переводит все чанки с учётом глоссария, стайлгайда, контекста мира и предыдущего чанка для сохранения связности.
  3. **Вычитка чанков (`proofreading`)**: Пошаговый качественный проход по каждому переведённому чанку.
  4. **Глобальная вычитка (`global_proofreading`)**: Финальная обработка собранного текста. LLM генерирует JSON-патчи; скрипт применяет их только при точном единичном совпадении.
* **Контекст мира (`world_info.md`)**: Поддержка файла с описанием вселенной произведения, который передаётся на этапы translation и proofreading.
* **Параллельная обработка**: Запускает несколько процессов `gemini-cli` одновременно.
* **Управление состоянием через SQLite**: Статусы чанков и глоссарий хранятся в локальных БД.
* **Интерактивный TUI**: Отображает прогресс выполнения и логи в реальном времени с помощью библиотеки `Rich`.
* **Отказоустойчивость**:
  * **Чекпоинты**: Сохранение статусов в БД позволяет пропускать уже выполненную работу при перезапуске.
  * **Возобновление**: Безопасно продолжает прерванный процесс с места остановки.
  * **Автоматические повторы**: Встроенная система ретраев (через `tenacity`) обрабатывает временные ошибки API.

## Архитектура проекта (Серии и Тома)

Проект использует иерархическую структуру, ориентированную на перевод серий книг, состоящих из нескольких томов.

### Корневая директория серии (Series Root)
В корне серии хранятся общие настройки и глобальный глоссарий, применимый ко всем томам:
* `book-translator.toml`: Главный файл конфигурации серии.
* `glossary.db`: Глобальная база данных SQLite с терминами для всей серии.
* `world_info.md`: Описание вселенной произведения (персонажи, сеттинг). Передаётся в промпты перевода и вычитки.
* `style_guide.md`: Правила стиля для перевода (автоматически копируется из бандлированных стайлгайдов при `init`).
* `prompts/`: Переопределения промптов для данной серии (опционально).

### Директория тома (Volume Directory)
Каждый том серии находится в отдельной поддиректории:
* `source/`: Исходные текстовые файлы (например, `chapter1.txt`, `prologue.txt`).
* `output/`: Готовые переведённые файлы (`.txt`, `.docx`, `.epub`).
* `.state/`: Локальная база данных состояния (`state.db`) и временные файлы для конкретного тома.

## Руководство пользователя

### 1. Предварительные требования

* Python 3.11+
* Утилита `gemini-cli` (должна быть установлена и авторизована через `gemini auth`)

### 2. Установка

Клонируйте репозиторий и установите зависимости:

```bash
pip install -e .
```

### 3. Использование CLI

Основная команда: `book-translator`.

#### Инициализация серии (`init`)
Создаёт структуру директорий для новой серии, конфигурационный файл `book-translator.toml`, файлы `world_info.md` и `style_guide.md` (автоматически подбирается стайлгайд под языковую пару), а также инициализирует `glossary.db`.

```bash
book-translator init "Название серии" --source-lang ja --target-lang ru
book-translator init "My Korean Novel" --source-lang ko --target-lang ru
book-translator init "Chinese Wuxia" --source-lang zh --target-lang ru
```

После инициализации **обязательно заполните** `world_info.md` и при необходимости отредактируйте `style_guide.md`.

#### Перевод файла (`translate`)
Запускает процесс перевода для конкретного файла. Скрипт автоматически найдёт конфигурацию серии в родительских директориях.

```bash
cd "Название серии"
book-translator translate volume-01/source/chapter1.txt
```

Дополнительные флаги:
* `--debug`: Включает подробное логирование.
* `--force`: Принудительно пересоздаёт чанки, удаляя старое состояние.
* `--resume`: Возобновляет прерванный перевод (поведение по умолчанию).
* `--dry-run`: Показывает план перевода без реальных вызовов API.
* `--stage`: Принудительно перезапускает с определённого этапа (`discovery`, `translation`, `proofreading`, `global_proofreading`).
* `--docx` / `--no-docx`: Включить или отключить конвертацию в `.docx`.
* `--epub` / `--no-epub`: Включить или отключить конвертацию в `.epub`.

#### Перевод всех томов (`translate-all`)
Последовательно переводит все тома серии.

```bash
book-translator translate-all
```

Также поддерживает флаги `--debug`, `--force`, `--resume`, `--dry-run`, `--stage`, `--docx`, `--no-docx`.

#### Управление глоссарием (`glossary`)
Просмотр, экспорт и импорт терминов глобального глоссария.

```bash
# Экспорт глоссария в TSV файл
book-translator glossary export --output terms.tsv

# Импорт глоссария из TSV файла
book-translator glossary import terms.tsv

# Просмотр всех терминов
book-translator glossary list
```

#### Статус перевода (`status`)
Отображает детализированный статус текущего процесса для серии или тома.

```bash
book-translator status
```

### 4. Конфигурация (`book-translator.toml`)

```toml
[series]
name = "My Series"
source_lang = "ja"
target_lang = "ru"

[gemini_cli]
model = "gemini-2.5-pro"

[splitter]
target_chunk_size = 600
max_part_chars = 800
min_chunk_size = 300

[workers]
max_concurrent = 50
```

* **`workers.max_concurrent`**: Количество одновременно запущенных процессов `gemini-cli`.
* **`splitter`**: Настройки разделения текста на чанки.
* **`gemini_cli.model`**: Модель Gemini (например, `"gemini-2.5-pro"`).

### 5. Стайлгайды

При инициализации серии автоматически копируется бандлированный стайлгайд для языковой пары:
* `ja_ru.md` — Японский→Русский (система Поливанова, суффиксы вежливости)
* `ko_ru.md` — Корейский→Русский (система Концевича)
* `zh_ru.md` — Китайский→Русский (система Палладия)
* `en_ru.md` — Английский→Русский
* `default.md` — Шаблон для других языковых пар

Для генерации кастомного стайлгайда под другую языковую пару используйте промпт из `docs/style_guide_prompt_template.md` с любой LLM.

### 6. Кастомизация промптов

Для переопределения промпта в конкретной серии создайте файл в `{series_root}/prompts/`:

```
{series_root}/
  prompts/
    translation.txt
    proofreading.txt
    global_proofreading.txt
    term_discovery.txt
```

Доступные плейсхолдеры: `{text}`, `{glossary}`, `{style_guide}`, `{world_info}`, `{previous_context}`, `{typography_rules}`, `{target_lang_name}`, `{source_lang_name}`.

### 7. Рабочий процесс

1. Инициализируйте серию: `book-translator init "my_series" --source-lang ja --target-lang ru`.
2. Заполните `world_info.md` (персонажи, сеттинг) и отредактируйте `style_guide.md`.
3. Поместите исходные файлы в `volume-01/source/`.
4. Запустите перевод: `book-translator translate volume-01/source/chapter1.txt`.
5. На этапе `discovery` подтвердите найденные термины в интерактивном TUI.
6. Следите за прогрессом translation → proofreading → global_proofreading.
7. Итоговые файлы появятся в `volume-01/output/`.

### Архитектура конвейера

Система построена на принципе конвейера из 4 этапов. Состояние надёжно хранится в SQLite.

* **CLI (`cli.py`)**: Обрабатывает команды пользователя.
* **Оркестратор (`orchestrator.py`)**: Управляет последовательностью этапов. Передаёт в промпты: текст, глоссарий, стайлгайд, world_info, типографические правила, языковые имена.
* **База данных (`db.py`)**: SQLite в режиме WAL. Глобальный глоссарий (`glossary.db`) + локальное состояние тома (`state.db`).
* **Модуль языков (`languages.py`)**: Маппинг кодов языков в имена (`ja` → `Japanese`) и типографические правила для каждого целевого языка.
* **Вычитка (`proofreader.py`)**: Применяет JSON-массив исправлений к тексту только при точном единичном совпадении.
* **Ограничитель запросов (`rate_limiter.py`)**: Строгое соблюдение 2 RPS.

### Жизненный цикл файла

1. **Разделение**: `chapter_splitter.py` разбивает исходный текст, сохраняет чанки в `state.db` со статусом `pending`.
2. **Этап 1 (Discovery)**: Параллельный поиск терминов в чанках; найденные предлагаются пользователю; статус → `discovery_done`.
3. **Этап 2 (Translation)**: Перевод чанков с глоссарием, стайлгайдом, world_info и контекстом предыдущего чанка; статус → `translation_done`.
4. **Этап 3 (Proofreading)**: Пошаговая вычитка каждого чанка; статус → `reading_done`.
5. **Сборка**: Оркестратор собирает черновой перевод из всех чанков.
6. **Этап 4 (Global Proofreading)**: Единый вызов LLM на весь текст; JSON-патчи применяются к тексту.
7. **Финал**: Итоговый файл сохраняется в `output/` (опционально конвертируется в `.docx`/`.epub`).
