# Book Translator (Script-Orchestrator)

Этот проект автоматизирует перевод больших текстовых файлов (ранобэ, манга, веб-романы) с любого языка на русский. Поддерживает два режима работы: **облачный** (через `gemini-cli`) и **локальный** (через [Ollama](https://ollama.com) — бесплатно, без лимитов, без интернета).

Поддерживаемые исходные языки: японский, корейский, китайский, английский и любой другой — достаточно создать стайлгайд.

## Основные возможности

* **Два LLM-бэкенда**: облачный Gemini или локальный Ollama — переключается одной строкой в конфиге.
* **Разные модели для разных задач**: каждый этап конвейера использует свою модель (маленькая быстрая для поиска терминов, большая мощная для перевода).
* **Точная настройка температуры**: отдельная температура для каждого этапа.
* **Поддержка любого исходного языка**: конвейер параметризован через языковые коды.
* **Интеллектуальное разделение глав**: автоматически разбивает большие файлы на семантически связанные чанки.
* **Четырёхэтапный конвейер**:
  1. **Поиск терминов (`discovery`)**: параллельно обрабатывает все чанки, находит новые термины. Предлагает добавить их в глоссарий.
  2. **Перевод (`translation`)**: переводит все чанки с учётом глоссария, стайлгайда, контекста мира и предыдущего чанка для связности.
  3. **Вычитка (`proofreading`)**: пошаговый качественный проход по каждому чанку.
  4. **Глобальная вычитка (`global_proofreading`)**: финальный проход по всей главе целиком. LLM генерирует JSON-патчи; скрипт применяет их только при точном единичном совпадении.
* **Управление состоянием через SQLite**: статусы чанков и глоссарий хранятся в локальных БД — перевод можно прерывать и возобновлять.
* **Интерактивный TUI**: прогресс выполнения и логи в реальном времени.
* **Автоматические повторы**: встроенная система ретраев обрабатывает временные ошибки.

├── glossary.db                     ← глобальный глоссарий (для всей серии)

```

### 2. Настройка окружения для локальных моделей (Ollama)

> Этот раздел нужен только если вы хотите использовать `backend = "ollama"`.
> Для работы с Gemini перейдите сразу к разделу 3.

#### Шаг 1: Установите Ollama

**macOS:**
```bash
# Способ 1: официальный установщик (рекомендуется)
# Скачайте с https://ollama.com/download и установите как обычное приложение.

# Способ 2: через Homebrew
brew install ollama
```

**Windows:**
```
Скачайте установщик OllamaSetup.exe с https://ollama.com/download
Запустите и следуйте инструкциям. Ollama автоматически добавится в автозапуск.
```

#### Шаг 2: Запустите Ollama-сервер

**macOS:**
```bash
# Если установили через приложение — запустите Ollama из Launchpad.
# Иконка появится в строке меню (верхний правый угол).

# Или из терминала:
ollama serve
```

**Windows:**
```
Ollama запускается автоматически при входе в систему.
Иконка появится в трее (нижний правый угол).
```

> После запуска Ollama слушает на `http://localhost:11434`. Проверьте в браузере — должна открыться страница с надписью "Ollama is running".

#### Шаг 3: Загрузите модели

Для каждой модели из конфига нужно выполнить `ollama pull`. Это нужно сделать только один раз — модели кэшируются на диске.

```bash
# Рекомендуемый набор для MacBook Pro M4 Pro 24GB:
ollama pull qwen3:8b          # ~5 GB  — для поиска терминов (быстрая)
ollama pull qwen3:30b-a3b     # ~18 GB — для перевода и вычитки (лучшая)
ollama pull qwen3:14b         # ~9 GB  — для глобальной вычитки
```

> **Сколько места нужно:** ~32 GB на диске для всех трёх моделей.
>
> **Как проверить, что модели загружены:**
> ```bash
> ollama list
> ```

#### Шаг 4: Проверьте, что всё работает

```bash
ollama run qwen3:8b "Привет! Переведи с японского: 勇者パーティー"
```

Если получили осмысленный ответ — всё готово.

### 5. Команды CLI

#### `init` — инициализация серии
```bash
book-translator init "Название серии" --source-lang ja --target-lang ru
book-translator init "My Korean Novel" --source-lang ko --target-lang ru
```

#### `translate` — перевод файла
```bash
book-translator translate volume-01/source/chapter1.txt
book-translator translate volume-01/source/chapter1.txt --force    # пересоздать с нуля
book-translator translate volume-01/source/chapter1.txt --resume   # продолжить с места остановки
book-translator translate volume-01/source/chapter1.txt --dry-run  # показать план без вызовов LLM
book-translator translate volume-01/source/chapter1.txt --stage translation  # перезапустить с этапа
```

#### `translate-all` — перевод всей серии
```bash
book-translator translate-all
book-translator translate-all --force
book-translator translate-all --stage proofreading
```

#### `glossary` — управление глоссарием
```bash
book-translator glossary export --output terms.tsv  # экспорт в таблицу
book-translator glossary import terms.tsv           # импорт из таблицы
book-translator glossary list                       # просмотр
```

#### `status` — статус перевода
```bash
book-translator status
```

### 6. Конфигурация (`book-translator.toml`)

Полный аннотированный конфиг со всеми доступными настройками:

```toml
# ─────────────────────────────────────────────────────────────────────────────
# Базовые настройки серии
# ─────────────────────────────────────────────────────────────────────────────
[series]
name        = "My Series"   # Название серии (обязательно)
source_lang = "ja"          # Исходный язык (2-буквенный ISO 639-1 код)
target_lang = "ru"          # Целевой язык

# ─────────────────────────────────────────────────────────────────────────────
# Выбор LLM-бэкенда
# ─────────────────────────────────────────────────────────────────────────────
[llm]
# "gemini" — облачный Gemini (нужен gemini-cli и интернет, есть лимиты)
# "ollama" — локальные модели через Ollama (нужен Ollama на компьютере,
#            бесплатно, без лимитов, работает офлайн)
backend = "ollama"

# Адрес Ollama-сервера. Не меняйте, если Ollama запущен на этом же компьютере.
ollama_url = "http://localhost:11434"

# ─────────────────────────────────────────────────────────────────────────────
# Модели для каждого этапа (используется только при backend = "ollama")
#
# Можно использовать разные модели для разных задач:
#   - Поиск терминов — задача простая (JSON с именами), достаточно маленькой модели
#   - Перевод и вычитка — сложные задачи, нужна лучшая модель
# ─────────────────────────────────────────────────────────────────────────────
[llm.models]
discovery           = "qwen3:8b"       # ~5 GB, быстрая — для поиска терминов
translation         = "qwen3:30b-a3b"  # ~18 GB — для перевода
proofreading        = "qwen3:30b-a3b"  # ~18 GB — для вычитки
global_proofreading = "qwen3:14b"      # ~9 GB  — для глобальной вычитки

# ─────────────────────────────────────────────────────────────────────────────
# Параметры генерации (используется только при backend = "ollama")
# ─────────────────────────────────────────────────────────────────────────────
[llm.options]

# Размер контекстного окна (в токенах).
# Промпт + глоссарий + текст чанка = ~3-4K токенов.
# 8192 достаточно для большинства случаев. Увеличьте до 16384, если глоссарий
# очень большой (200+ терминов) или style_guide длинный.
num_ctx = 8192

# Базовая "температура" для всех этапов (переопределяется в [llm.options.stage_temperature]).
# 0.0 = полностью детерминированный вывод (одинаковый результат каждый раз)
# 1.0 = максимальная случайность
# Для перевода обычно 0.2–0.5
temperature = 0.3

# Режим "размышления" у моделей Qwen3.
#
# Qwen3 умеет думать вслух перед ответом (<think>...</think>) — это может
# улучшить качество на сложных задачах, но:
#   - Значительно замедляет генерацию (добавляет 200-500 лишних токенов)
#   - Thinking-токены могут "протечь" в переведённый текст
#   - JSON-выход становится менее надёжным
#
# false = выключить (рекомендуется для перевода — быстрее и чище)
# true  = включить (для экспериментов с качеством)
think = false

# ─────────────────────────────────────────────────────────────────────────────
# Температура для каждого этапа отдельно
# (переопределяет базовое значение temperature выше)
#
# Почему разные значения:
#   discovery: нужен точный детерминированный JSON → низкая температура
#   translation: лёгкая вариативность делает текст живее → чуть выше
#   proofreading: исправляем стиль, но не изобретаем новый текст → умеренно
#   global_proofreading: снова JSON-патчи → низкая температура
# ─────────────────────────────────────────────────────────────────────────────
[llm.options.stage_temperature]
discovery           = 0.1
translation         = 0.4
proofreading        = 0.3
global_proofreading = 0.1

# ─────────────────────────────────────────────────────────────────────────────
# Настройки Gemini CLI (используется только при backend = "gemini")
# ─────────────────────────────────────────────────────────────────────────────
[gemini_cli]
model                        = "gemini-2.5-pro"
worker_timeout_seconds       = 120   # таймаут одного вызова LLM
proofreading_timeout_seconds = 300   # таймаут глобальной вычитки (вся глава)

# ─────────────────────────────────────────────────────────────────────────────
# Параллелизм
# ─────────────────────────────────────────────────────────────────────────────
[workers]
# Количество одновременных вызовов LLM.
#
# Для Ollama: 1–4
#   Локальный GPU обрабатывает ОДНУ задачу за раз. Большее значение просто
#   создаёт очередь в Ollama — быстрее не станет, но памяти потребует больше.
#   Значение 2–4 даёт небольшой прирост за счёт перекрытия I/O и вычислений.
#
# Для Gemini: 10–50
#   Облако справляется с параллелизмом, высокое значение ускоряет перевод.
max_concurrent = 2

# Запросов в секунду.
# Для Ollama: высокое значение (сервер сам управляет очередью)
# Для Gemini: 2.0 (лимит бесплатного тарифа)
max_rps = 100.0

# ─────────────────────────────────────────────────────────────────────────────
# Разбивка текста на чанки
# ─────────────────────────────────────────────────────────────────────────────
[splitter]
target_chunk_size = 600  # целевой размер чанка в символах
max_part_chars    = 800  # максимальный размер части
min_chunk_size    = 300  # минимальный размер чанка

# ─────────────────────────────────────────────────────────────────────────────
# Повторные попытки при ошибках LLM
# ─────────────────────────────────────────────────────────────────────────────
[retry]
max_attempts     = 3   # сколько раз повторить при ошибке
wait_min_seconds = 4   # минимальная пауза между попытками
wait_max_seconds = 10  # максимальная пауза
```

> **Минимальный конфиг для Ollama** (только обязательные настройки):
> ```toml
> [series]
> name = "My Series"
> source_lang = "ja"
> target_lang = "ru"
>
> [llm]
> backend = "ollama"
>
> [workers]
> max_concurrent = 2
> max_rps = 100.0
> ```
> Все остальные параметры заполнятся дефолтными значениями.

### 7. Стайлгайды

При инициализации серии автоматически копируется бандлированный стайлгайд:
* `ja_ru.md` — Японский→Русский (система Поливанова, суффиксы вежливости)
* `ko_ru.md` — Корейский→Русский (система Концевича)
* `zh_ru.md` — Китайский→Русский (система Палладия)
* `en_ru.md` — Английский→Русский
* `default.md` — шаблон для других языковых пар

### 8. Кастомизация промптов

Для переопределения промпта создайте файл в `{series_root}/prompts/`:

```
МояСерия/
  prompts/
    translation.txt          ← заменит дефолтный промпт перевода
    proofreading.txt         ← заменит промпт вычитки
    global_proofreading.txt
    term_discovery.txt
```

Пользовательские промпты имеют наивысший приоритет и используются для **любого** бэкенда (и Ollama, и Gemini).

Доступные плейсхолдеры: `{text}`, `{glossary}`, `{style_guide}`, `{world_info}`, `{previous_context}`, `{typography_rules}`, `{target_lang_name}`, `{source_lang_name}`.

**Встроенные промпты:**
- `data/prompts/` — полные детализированные промпты для Gemini
- `data/prompts/local/` — упрощённые промпты для локальных моделей (используются автоматически при `backend = "ollama"`)

---

### 9. Рабочий процесс

1. Инициализируйте серию: `book-translator init "my_series" --source-lang ja --target-lang ru`
2. Откройте `book-translator.toml`, выберите `backend = "ollama"` или `backend = "gemini"`
3. Заполните `world_info.md` (персонажи, сеттинг) и отредактируйте `style_guide.md`
4. Загрузите нужные модели: `ollama pull qwen3:30b-a3b` (для Ollama)
5. Поместите исходные файлы в `volume-01/source/`
6. Запустите: `book-translator translate volume-01/source/chapter1.txt`
7. На этапе `discovery` подтвердите найденные термины в TUI
8. Следите за прогрессом: discovery → translation → proofreading → global_proofreading
9. Итоговые файлы появятся в `volume-01/output/`

## Руководство для разработчиков

### Структура проекта

```
src/book_translator/
├── cli.py / commands/      — CLI и подкоманды
├── orchestrator.py         — управление конвейером
├── llm_runner.py           — вызовы LLM (Gemini subprocess + Ollama HTTP)
├── discovery.py            — загрузка конфига, поиск корня серии
├── db.py                   — все SQLite-операции
├── path_resolver.py        — разрешение путей и промптов
├── default_prompts.py      — встроенные промпты (cloud + local)
├── chapter_splitter.py     — семантическое разделение на чанки
├── term_collector.py       — разбор JSON от LLM, TSV-воркфлоу
├── proofreader.py          — применение JSON-патчей
├── rate_limiter.py         — ограничитель запросов
├── languages.py            — коды языков, типографика
├── tui.py                  — Rich-прогресс и статус
└── data/
    ├── prompts/            — промпты для Gemini (полные)
    └── prompts/local/      — промпты для Ollama (упрощённые)
```

### Ключевые модули

| Модуль | Ответственность |
|---|---|
| `orchestrator.py` | Управление последовательностью конвейера, per-stage модели |
| `llm_runner.py` | `run_gemini()`, `run_ollama()`, `run_llm()` диспетчер |
| `discovery.py` | Загрузка конфигурации с дефолтами и валидацией |
| `path_resolver.py` | Разрешение промптов с учётом бэкенда |
| `db.py` | SQLite: `glossary.db` (серия) + `chunks.db` (том) |
