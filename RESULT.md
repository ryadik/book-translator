# Анализ системы промптов book-translator

## 1. АНАЛИЗ ПРОМПТОВ

### 1.1. `translation` — Промпт перевода

**Назначение:** Перевод исходного текста (японский) на русский литературный язык.

**Структура (8 секций):**
1. Priming & Keywords — набор ключевых слов для настройки "режима" модели
2. Core Objective — роль (профессиональный литературный переводчик), режим (stateless)
3. Data Layer — инъекция `{glossary}` и `{style_guide}` через XML-теги
4. Execution Anchors (A-D) — четыре группы строгих правил:
   - **A: Content Physics** — сохранение объёма, фактов, сенсорных деталей, психологии
   - **B: Stylistic Vector** — русская литературная норма, инверсии, звукоподражания, культурная адаптация
   - **C: Text Architecture** — реструктуризация абзацев, использование оператора звука (`─`)
   - **D: Binary Text Physics & Typography** — бинарная модель `[SOUND]`/`[SILENCE]`, правила тире, кавычек, нестандартных коммуникаций
5. Task — задание на перевод с ограничениями
6. Source Fragment — `{previous_context}` + `{text}` в XML-тегах
7. Mandatory Output Filters — пять финальных проверок
8. Output Instruction — "только сырой текст, без markdown"

**Оценка качества:** Очень высокое. Промпт детально проработан, покрывает все edge-кейсы типографики, имеет чёткую бинарную модель обработки тире. Самый сильный промпт в системе.

---

### 1.2. `proofreading` — Промпт вычитки (per-chunk)

**Назначение:** Литературная полировка уже переведённого русского текста.

**Структура:** Идентична translation, за исключением:
- Anchor A заменён на Stylistic Refinement (обогащение лексики, устранение канцелярита, динамизация глаголов)
- Anchor B — Canon Compliance (проверка глоссария и style guide)
- Anchors C и D — **полная копия** из translation промпта

**Оценка качества:** Хорошее. Корректно разделяет задачи — редактор не переводит, а полирует. Однако дублирование Anchor D целиком (~60 строк) увеличивает объём промпта без необходимости.

---

### 1.3. `term_discovery` — Промпт обнаружения терминов

**Назначение:** Извлечение новых терминов (имена, локации, уникальные понятия) из исходного текста.

**Структура (5 секций):**
1. Роль и цель — эксперт-терминолог
2. Принципы анализа — что извлекать, что игнорировать (с примерами)
3. Рабочий процесс — три шага (извлечение → фильтрация → структуризация)
4. Данные — `{glossary}` и `{text}` (в markdown code fences)
5. Формат вывода — JSON с примером

**Оценка качества:** Хорошее, но с проблемами (см. ниже). Хорошо определяет границы между шумом и значимыми терминами. Примеры конкретные и полезные.

---

### 1.4. `global_proofreading` — Промпт глобальной вычитки

**Назначение:** Обзор целой главы (все чанки) для выявления межчанковых несоответствий.

**Структура:** Минимальная — роль, описание формата ввода, правила для JSON-диффов, пример.

**Оценка качества:** Слабое. Это самый слабый промпт в системе. Причины:
- Нет глоссария, стиль-гайда, типографических правил
- Нет понимания контекста мира/персонажей
- Говорит `content_en` (промпт писался для английского), код патчит это через `.replace("content_en", "content_source")`
- Нет инструкций по стилю, тональности, или литературным стандартам

---

### 1.5. `style_guide.md` (data/)

**Назначение:** Правила адаптации с японского на русский (пунктуация, имена, суффиксы, структура).

**Оценка качества:** Отличное. Очень детально прописаны правила обработки японских пунктуационных конструкций (`「」`, `『』`, `（）`, `……`, `〜`), система адаптации именных суффиксов с примерами, правила формирования абзацев.

---

### 1.6. `world_info.md` (data/)

**Назначение:** Описание вселенной (DanMachi) — жанр, сеттинг, система, главный герой, тональность.

**Оценка качества:** Хорошее для конкретной книги. Даёт LLM контекст для принятия решений о тоне и стиле.

## 2. АНАЛИЗ ПРОМПТ-СИСТЕМЫ В КОДЕ

### 2.1. Хранение промптов

| Компонент | Путь |
|---|---|
| Бандлированные промпты | `src/book_translator/default_prompts.py` — 4 константы + реестр `PROMPTS` |
| Файлы-оверрайды | `{series_root}/prompts/{name}.txt` — приоритет над бандлированными |
| Резолвер | `path_resolver.resolve_prompt()` — проверяет оверрайд → fallback на бандлированный |

### 2.2. Загрузка данных

```
orchestrator.run_translation_process()
├── Промпты: path_resolver.resolve_prompt() × 4
├── Глоссарий: db.get_terms() → json.dumps() → JSON-массив [{term_source, term_target, ...}]
├── Style guide: series_paths.style_guide.read_text() → сырой markdown
└── World info: series_paths.world_info.read_text() → ЗАГРУЖЕН, НО НЕ ИСПОЛЬЗУЕТСЯ
```

### 2.3. Подстановка переменных

Механизм — цепочка `.replace()` в `_run_single_worker()`:
```python
final_prompt = prompt_template \
    .replace('{text}', chunk_content) \
    .replace('{glossary}', glossary_str) \
    .replace('{style_guide}', style_guide_str) \
    .replace('{previous_context}', previous_context)
```

**Доступные плейсхолдеры:** `{text}`, `{glossary}`, `{style_guide}`, `{previous_context}`

**Использование по этапам:**

| Этап | `{text}` | `{glossary}` | `{style_guide}` | `{previous_context}` |
|---|---|---|---|---|
| Discovery | content_source | да | нет (пусто) | нет (пусто) |
| Translation | content_source | да | да | content_source предыдущего чанка |
| Proofreading | content_target | да | да | content_target предыдущего чанка |
| Global Proofreading | Все чанки конкатенированы напрямую (не через `.replace()`) | нет | нет | нет |

### 2.4. Вызов LLM

```python
command = ['gemini', '-m', model_name, '-p', final_prompt, '--output-format', format]
subprocess.run(command, stdout=PIPE, stderr=PIPE, timeout=worker_timeout)
```

- Discovery: `--output-format json`
- Translation/Proofreading: `--output-format text`
- Global proofreading: `--output-format json`

### 2.5. Пайплайн по этапам

```
[Разделение на чанки]
        ↓
[Discovery] — параллельно, JSON-выход → term_collector → TSV-подтверждение → glossary.db
        ↓
[Translation] — параллельно, текстовый выход → content_target
        ↓
[Proofreading] — параллельно, текстовый выход → content_target (перезапись)
        ↓
[Global Proofreading] — один вызов, JSON-диффы → apply_diffs → content_target
        ↓
[Сборка] → .txt → опционально .docx
```

## 3. ПОДРОБНЫЙ ОТЧЁТ: ВЫЯВЛЕННЫЕ ПРОБЛЕМЫ

### 3.1. КРИТИЧЕСКИЕ БАГИ

#### BUG-1: Дублирование `{text}` в промптах translation и proofreading

**Файл:** `default_prompts.py`, строки 135-139 (translation) и 380-384 (proofreading)

**Проблема:** Секция "SOURCE FRAGMENT" содержит:
```
<source_text>
{text}
</source_text>
{text}
</source_text>
```

Текст инъектируется **дважды**: один раз внутри корректного XML-тега, и второй раз — без открывающего тега. После `.replace('{text}', ...)` LLM получает:
```
<source_text>
[весь текст чанка]
</source_text>
[весь текст чанка — повтор]
</source_text>
```

**Последствия:** Удвоение входных токенов, путаница модели — она может перевести текст дважды или обработать "хвост" как часть инструкции.

**Есть в файлах-оверрайдах:** Да, та же ошибка в `prompts/translation.txt` и `prompts/proofreading.txt`.

---

#### BUG-2: `proofreader.py` использует ключ `content_ru`, а оркестратор — `content_target`

**Файл:** `proofreader.py`, строки 35 и 42

**Проблема:** Функция `apply_diffs()` ищет:
```python
content = str(chunk.get("content_ru", ""))  # всегда пустая строка!
```
и записывает:
```python
chunk["content_ru"] = content.replace(find_str, replace_str)
```

Но оркестратор передаёт чанки с ключом `content_target`, не `content_ru`. В результате:
- `content` всегда `""` (пустая строка)
- `find_str` никогда не найдена (0 вхождений)
- Все диффы пропускаются с warning `'find' string not found`
- `chunk["content_ru"]` записывается, но оркестратор потом читает `chunk['content_target']` — неизменённое

**Последствия:** Глобальная вычитка (Этап 3.5) **полностью нефункциональна**. LLM вызывается, тратит токены, генерирует правки — но ни одна правка не применяется.

---

#### BUG-3: `global_proofreading` промпт использует `content_en`/`content_ru`, оркестратор патчит через `.replace()`

**Файл:** `orchestrator.py`, строка 145

**Проблема:** Промпт написан с `content_en` и `content_ru`, но данные приходят как `content_source` и `content_target`. Код делает:
```python
final_prompt = final_prompt.replace("content_en", "content_source")
```

Но **не заменяет** `content_ru` на `content_target`. LLM видит метки `content_ru` в данных, но фактические ключи в форматированных чанках — `content_target`. При этом форматирование чанков (строка 141):
```python
f"content_source: {chunk['content_source']}\ncontent_target: {chunk['content_target']}"
```
уже использует `content_source`/`content_target`. Значит в финальном промпте у данных ключи `content_source`/`content_target`, а в инструкции — `content_source`/`content_ru`. Несоответствие.

### 3.2. СУЩЕСТВЕННЫЕ ПРОБЛЕМЫ

#### ISSUE-1: `world_info.md` загружается, но не используется

**Файл:** `orchestrator.py`, строка 318

```python
world_info_content = series_paths.world_info.read_text(encoding='utf-8') if series_paths.world_info else ''
```

Переменная `world_info_content` загружена, но **нигде не передаётся** в промпты. Нет плейсхолдера `{world_info}` ни в одном шаблоне. Dry-run выводит путь к world_info, но данные мертвы.

**Последствия:** Описание мира, жанра, персонажей, тональности — недоступно модели при переводе. Модель не знает, что переводит фэнтези, не знает персонажей, не может корректно выбрать тон.

---

#### ISSUE-2: `term_discovery` промпт захардкожен под DanMachi

**Файл:** `default_prompts.py`, строка 169

```
Ты — Эксперт-Терминолог и Аналитик Мира, специализирующийся на вселенной "DanMachi".
```

**Последствия:** При использовании для другой серии (книги) промпт будет сообщать модели, что она работает с DanMachi, даже если переводится совершенно другое произведение. Это может привести к некорректной категоризации терминов.

---

#### ISSUE-3: Двойные фигурные скобки `{{` в JSON-примере term_discovery

**Файл:** `default_prompts.py`, строки 224-262

JSON-пример в промпте использует `{{` и `}}` вместо `{` и `}`:
```json
{{
  "characters": {{
    "aizu_varenshutain": {{
```

Поскольку подстановка использует `.replace()`, а не f-string, `{{` не раскрывается и передаётся модели как есть. Модель получает невалидный JSON-пример.

**Последствия:** Модель видит `{{...}}` вместо `{...}`, что может привести к генерации невалидного JSON в ответе (хотя на практике модели обычно справляются).

---

#### ISSUE-4: `global_proofreading` промпт — качественный провал

**Сравнение с другими промптами:**

| Аспект | translation/proofreading | global_proofreading |
|---|---|---|
| Глоссарий | Да | Нет |
| Style guide | Да | Нет |
| Типографические правила | ~60 строк детальных правил | Нет |
| Литературные инструкции | Да | Нет |
| Контекст мира | Через style_guide | Нет |
| Язык инструкций | Английский (профессиональный) | Английский (базовый) |
| Объём | ~160 строк | ~45 строк |

Это означает, что глобальный редактор может вносить правки, нарушающие типографические правила (заменить `─` на `—`), нарушить глоссарий (написать имя иначе), или ухудшить стиль.

---

#### ISSUE-5: Противоречие между style_guide.md и промптами по тире

**Style guide (`data/style_guide.md`):**
> Тире в диалогах: Использовать длинное тире (—).
> `「こんにちは」と彼は言った。` -> `— Привет, — сказал он.`

**Промпты translation/proofreading (ANCHOR D):**
> The em-dash (`—`) is forbidden everywhere.
> Use `─` (U+2500) as the sole separator between [SOUND] and [SILENCE].

Три разных символа, два противоречащих правила:
- Style guide говорит использовать `—` (em-dash, U+2014)
- Промпт запрещает `—` и требует `─` (box drawing, U+2500)

Модель получает оба документа одновременно и должна выбирать. Это прямой конфликт.

---

#### ISSUE-6: Закрывающий тег `</output>` без открывающего

Все четыре промпта заканчиваются тегом `</output>`, но ни один не содержит открывающего `<output>`. Модель может интерпретировать это как:
- Закрытие несуществующего тега (игнорировать)
- Сигнал остановки генерации
- Часть разметки, которую нужно включить в вывод

### 3.3. МЕЛКИЕ ЗАМЕЧАНИЯ

#### NOTE-1: Предыдущий контекст для перевода — исходный текст, а не перевод

При translation, `{previous_context}` заполняется из `content_source` предыдущего чанка (строка 424). Модель видит предыдущий чанк на японском, а не его перевод. Это может быть осознанным решением (чтобы дать контекст на языке оригинала), но лишает переводчика информации о том, КАК был переведён предыдущий фрагмент (какие стилистические решения были приняты, какие имена использованы).

#### NOTE-2: Discovery не получает style_guide

На этапе Discovery `style_guide_str` передаётся пустым (не задан в `_run_workers_pooled`). Это означает, что при обнаружении терминов модель не знает правил транскрипции имён (система Поливанова, исключения из глоссария), что может привести к некорректным предложениям по транскрипции.

#### NOTE-3: Промпт proofreading дублирует всю типографическую секцию

Anchor D в `proofreading` — это точная копия Anchor D из `translation` (~60 строк). При изменении правил типографики нужно обновлять оба промпта. Это нарушение DRY-принципа.

#### NOTE-4: `_style_guide.md` и `_glossary_old.json` — мусорные файлы

Файл `data/_style_guide.md` содержит только текст "Style guide". Файл `data/_glossary_old.json` — видимо, старая версия. Оба не используются в пайплайне.

**Общая оценка:** Промпты translation и proofreading проработаны на высоком уровне. Типографическая система (`[SOUND]`/`[SILENCE]`, бинарная логика тире) — оригинальная и эффективная. Style guide для японского — отличный. Однако система содержит три критических бага, которые делают глобальную вычитку полностью нерабочей и удваивают входные токены на каждом вызове перевода/вычитки. Global proofreading промпт нуждается в серьёзной переработке. World info загружается впустую.
